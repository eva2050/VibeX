# 中文 X Post Skill v1.3：本人信号优先设计

## 背景

v1.2 使用 40 条近期中文 X 正反语料重建信号诊断和反模式判断，但第二轮 15 条人工审核仍未达到发布预期。

失败不再主要表现为明显的“底层逻辑”“真正重要的是”等说教词，而是更隐蔽的二手资讯编辑感：系统把原作者的推文换一种方式说出来，沿用统一的“事实—转折—小结”结构，并把原作者的测试、出行和产品体验错误地写成用户第一人称。内容虽然更顺，却没有属于用户自己的新增信息，也没有非发不可的理由。

v1.3 不继续增加文风规则。核心变化是先验证内容所有权和本人信号，再决定是否成稿。

## 目标

- Studio 根据用户本人提供的经历、动作、数据、判断和产品细节生成一篇正文。
- 中文 X 语料只训练判断力和失败拦截，不直接提供当前正文的事实、经历、观点或句子。
- 素材不足时忠实写短或拒绝扩写，不用热点、方法论和假经历填空。
- 先用 VibeX 最近的真实开发记录生成第三轮 15 条审核内容，人工达标后再修改产品代码。
- 让人工审核回答“我愿不愿意用自己的账号发布”，而不是“这段话是否写得通顺”。

## 非目标

- 不追求复刻某一位中文 X 作者的口吻。
- 不把高播放量当成内容质量或产品效果的证明。
- 不在运行时抓取热门推文替用户决定话题。
- 不增加 Studio 的输入步骤、风格设置或候选选择器。
- 不先上线 v1.3 代码再依赖产品内数据判断方向。

## 核心原则

### 1. 本人信号是成稿的前提

每条可发布正文至少包含一个属于用户的信号：

- 用户亲自做过的动作；
- 用户观察到的具体产品行为或摩擦；
- 用户自己的数据、结果或失败；
- 用户明确说出的判断、疑问或情绪；
- 用户正在推进的产品决策及其理由。

如果输入只有外部新闻、别人的经历或抽象话题，Skill 可以整理来源或帮助用户提出问题，但不能把它伪装成本人经历。

### 2. 先确认“谁做的”，再决定人称

任何第一人称动作都必须能回指到用户输入或用户授权的第一方记录。引用内容中的“我测试了”“我去了”“我发现”属于原作者，不能转移给用户。

素材归属分为三类：

| 归属 | 允许的写法 |
| --- | --- |
| 用户本人 | 可以使用第一人称并保留动作、结果和情绪 |
| 明确外部来源 | 使用来源归属，不改写为用户亲历 |
| 无法判断 | 不使用第一人称，不补经历 |

### 3. 语料只做评审，不做内容供给

40 条中文 X 语料继续保留，但职责收窄为：

- 识别说教、标题党、软广、伪反差、伪步骤和二手资讯腔；
- 判断成稿是否过于完整、平滑和统一；
- 提供跨作者的表达边界，不提供当前主题。

运行时提示不得向模型暴露可被复述的语料摘要。策略卡不再要求固定的“事实—判断—停止”形状，只描述必须保留的证据和禁止增加的内容。

### 4. 一条内容只需要一个发布理由

发布理由必须来自素材本身，例如：

- 今天做了一个具体决定；
- 一个测试结果和预期不同；
- 产品出现一个真实摩擦；
- 用户反馈改变了设计；
- 失败暴露了原先的错误假设；
- 一个正在发生但尚未想清楚的问题。

正文可以停在观察、问题或情绪，不强制得出结论。

## 第三轮实验：先验证内容，不改产品

### 第一方信号来源

第三轮只使用能在 VibeX 仓库或本次产品讨论中核对的记录：

- 最近的 Git 提交及对应代码差异；
- 已确认的产品决策，例如 Studio 单稿、Post / Article、Reply 留在 X 页面；
- v1.1 和 v1.2 的真实人工审核结果；
- Auto、Studio、Rewrite、Posts 复盘和 Loop 机制的真实取舍；
- 已经发生的中断、失败、返工和版本变化。

每条审核内容都附一行“事实依据”，但发布正文不显示内部解释。

### 15 条内容的分布

- 5 条产品决策或取舍；
- 4 条失败和返工；
- 3 条具体开发观察；
- 2 条尚未解决的问题；
- 1 条短情绪或现场片段。

内容长度不设统一区间，允许一句话、两段或较长记录。连续三条不得使用相同段落骨架。

### 审核方式

审核者一次性阅读 15 条，只反馈：

- 整体分；
- 愿意直接发布的编号；
- 共同问题。

第三轮进入产品实现的门槛：

- 至少 8/15 愿意直接发布；
- 整体分不低于 75；
- 不能出现冒用外部作者第一人称的内容；
- 不能因相同结构产生明显的批量生成感。

未达到门槛时继续调整内容规则和审核集，不修改 Studio 或 Auto 运行代码。

## 达标后的产品设计

### 诊断输出

中文 Post Skill 的分析结果新增：

- `ownership`: `first_party | attributed_external | unknown`
- `firstPartySignals`: 用户本人的动作、数据、产品细节、判断或情绪
- `externalSignals`: 带来源的外部事实
- `publishReason`: 当前素材为什么值得发
- `allowedPerspective`: 允许使用的叙述人称

### 生成规则

1. 先锁定第一方信号和归属。
2. 只选择一个发布理由，不选择固定文章结构。
3. 保留用户原话中最有个性的词、停顿和不完整表达。
4. 草稿已经表达完信号时立即停止。
5. 没有第一方信号时，不伪造用户经历；如果用户只要求整理外部信息，必须明确来源归属。

### 独立评审

新增硬失败：

- `borrowed_first_person`：把外部来源经历写成用户经历；
- `no_first_party_signal`：声称是用户原创正文，但没有用户本人信号；
- `source_paraphrase_only`：正文只是对外部来源换词复述；
- `uniform_micro_essay`：固定使用事实、转折、小结的微型作文骨架；
- `invented_publish_reason`：素材没有提供却补出宏观意义或行动建议。

确定性检查、数字与实体边界、伪步骤和内容农场词拦截继续保留。

## 测试设计

### 所有权测试

- 引用推文中的第一人称不得成为生成者第一人称。
- 用户明确描述自己的测试时允许使用“我”。
- 无法确认归属时必须降级为来源归属或非第一人称。

### 信号测试

- 强第一方素材必须保留至少一个本人动作和一个结果或摩擦。
- 只有外部新闻的输入不得被判为第一方内容。
- 薄输入生成长度不得明显超过原始信息量。

### 多样性测试

- 审核集 15 条不得共享统一的三段式结构。
- 不得连续重复“但”“不是……而是……”“问题在于”等转折骨架。
- 不用词面随机化掩盖相同逻辑结构。

### 回归测试

- Studio 仍只返回一篇正文。
- Article 流程和 X 页面 Reply 入口不受影响。
- Auto 的安全边界不变，不恢复为默认使用路径。
- 原有事实忠实度、语言和生成归因测试继续通过。

## 示例目标

第一方事实：团队为中文 X Skill 收集了 40 条热门推文，v1.2 第二轮内容更顺但仍未通过人工审核。

可接受正文：

> 我给中文 X 写作 Skill 喂了 40 条热门推文。
>
> 第二轮出来的东西更顺了，也更像别人了。
>
> 问题可能就在这里：AI 学会了内容长什么样，却没有任何东西非说不可。

这条正文的价值来自真实返工和矛盾结果，不来自模仿热门作者或补一套写作方法论。

## 发布顺序

1. 从 VibeX 第一方记录中提取 15 个可核对信号。
2. 生成第三轮 15 条审核内容并进行人工验收。
3. 未达标时只迭代内容规则。
4. 达标后为所有权诊断和硬失败编写回归测试。
5. 实现 v1.3 Skill，并接入 Studio 和 Auto 的现有接口。
6. 跑完整测试后提交和推送。
