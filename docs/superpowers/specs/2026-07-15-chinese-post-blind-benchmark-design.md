# 中文 X Post Skill 批量盲测设计

## 目标

建立一套可重复、可中断续跑的真实 A/B 测试，回答一个核心问题：在使用相同模型、账号上下文和生产链路时，`zh-x-post@1.0.0` 是否比当前 Studio 生成效果更好。

用户扮演内容主编和审核者，不负责逐项决定技术参数。系统一次性交付 10 条内容供集中审核，用户可以只给总分和少量批注；Codex 根据反馈与客观指标继续修改 Skill、生成下一批并复测。除方向冲突、外部授权或不可逆操作外，中间不反复询问。

## 本阶段范围

包含：

- 24 条固定中文素材，6 类内容各 4 条；
- 当前 Studio 与中文 Skill Studio 的完整生产链路对比；
- 每条内容进行两次 A/B 顺序互换的模型盲评；
- 每轮集中展示 10 条 Skill 内容给用户审核；
- 用户用总分、优秀样本、问题样本和一句批注完成审核；
- 保留 14 条隐藏验证样本，防止只针对审核样本调参；
- 事实安全指标、分内容类型数据和可复现报告；
- 通过全部门槛后，只开启中文 Studio Post Skill。

不包含：

- 将测试内容发布到 X；
- 使用浏览量、点赞、回复或转发作为本轮成功指标；
- 开启中文 Auto Post；
- Article Skill 或 Reply Skill；
- 导出 API Key 或将测试素材发送给额外服务。

## 成功标准

Skill 只有同时满足以下条件才算通过：

- 在模型双盲 A/B 中，非平局样本的 Skill 胜率不低于 65%；
- 24 条中至少 18 条得到非平局结论；
- Skill 输出的原意与主张保留率不低于 95%；
- 新增事实数量为 0；
- 模板腔命中率不高于 10%；
- 24 条素材的当前版和 Skill 版都得到有效输出；
- 用户对最近一批 10 条内容明确给出通过，或批次总分达到 85 分；
- 两个对比组使用完全相同的模型、语言、账号上下文和生成配置。

盲选胜率不能抵消事实错误。任何缺失输出、未完成评审或中断状态都不能判定通过。

## 测试数据

从 `benchmarks/chinesePostFixtures.js` 中固定选择 24 条素材，每类 4 条：

- 产品观察；
- 工具体验；
- Build in Public；
- 失败复盘；
- 行业观点；
- 工作流框架。

具体素材 ID 固定在 benchmark 版本清单中，不随运行随机变化。报告同时保存 benchmark 版本、Skill 版本和提交号，保证不同版本之间可以公平对比。

## A/B 生成链路

每条素材分别运行两个组：

1. 当前组：明确关闭中文 Skill，使用现有通用 Studio 改写链路。
2. Skill 组：明确加载 `zh-x-post@1.0.0`，使用相同 Studio 链路。

两个组使用同一个扩展内已配置的 Provider、模型、语言、账号画像、表现记忆、用户偏好和素材。API Key 只在现有 Service Worker 内参与调用，不进入报告、不显示在页面，也不导出。

每组都真实运行 3 个候选生成、独立候选评审，以及质量不通过时最多 1 次修复。不能使用单条简化 Prompt 代替生产链路。

测试按素材顺序逐条运行，每完成一个组和一条素材就立即保存检查点。关闭页面、浏览器重启或短时断网后可以继续，不重复消耗已完成的调用。

## 模型双盲评审

每条素材得到两个最终输出后，用固定随机种子生成匿名 A/B 顺序：

- 第一次评审按 A/B 顺序展示；
- 第二次评审交换顺序；
- 评审模型看不到 Skill 名称、版本、策略、Prompt 和候选元数据。

盲评维度包括：原意忠实度、信息增量、自然中文 X 表达、首句吸引力、读者价值和模板风险。

两次评审指向同一个真实版本时，记为稳定胜负；两次结论不一致时，记为模型平局。最终模型指标为：

`Skill 胜数 /（Skill 胜数 + 当前版胜数）`

平局单独报告，不进入胜率分母。若非平局少于 18 条，本轮不能通过。

## 用户批量审核

用户每轮只审核一个包含 10 条 Skill 输出的批次，不逐条弹问。

10 条内容的选择规则：

1. 优先覆盖模型判断最不确定或 Skill 表现最弱的内容；
2. 6 类内容至少各出现 1 条；
3. 剩余名额优先选择事实风险、模板风险或结构重复较高的样本。

审核页面一次性展示完整 10 条内容。用户可以使用非常简短的反馈格式，例如：

`整体 70 分；2、6 最好；4、9 AI 味重；观点不够具体。`

最低输入只需要一个 0–100 的总分。优秀编号、问题编号和文字批注都是可选项。系统保存原始反馈，并将其归纳为：Hook、具体性、自然度、事实忠实度、结构、节奏、信息增量和账号匹配等问题标签。

用户不需要选择技术方案，也不需要确认每一次 Skill 修改。总分低于 85 或用户未明确通过时，Codex直接进入下一轮优化。

## 迭代方式

每轮结束后，Codex一次性给出：

- 本轮 10 条内容；
- 当前模型盲测胜率与安全指标；
- 相比上一轮的变化；
- 本轮最明显的 2–3 个问题。

用户集中打分后，Codex执行：

1. 把反馈映射到具体失败模式；
2. 修改版本化 Skill，而不是覆盖历史版本；
3. 对 10 条审核样本复测，确认反馈问题是否改善；
4. 对 14 条隐藏样本复测，检查是否过拟合；
5. 输出下一批 10 条和量化变化。

隐藏验证集在最终通过前不展示具体内容，只展示汇总数据。若审核集提升但隐藏集下降，则判定为过拟合，不能开启 Studio。

## 状态与存储

盲测数据使用独立、版本化的 `chrome.storage.local` 键，与 generation sessions、Posts、Loop、账号记忆和发布归因完全隔离。

每次运行保存：

- 运行 ID、状态、benchmark 版本、Skill ID/版本、应用提交号；
- Provider 与模型名称，不保存额外 API Key 副本；
- 24 条素材 ID、匿名排序种子和审核集/隐藏集划分；
- 当前版输出、Skill 输出和各自生产评审结果；
- 两次匿名模型盲评；
- 用户原始总分、可选编号和批注；
- 自动归纳的问题标签；
- 安全指标、胜率、分类型结果、发布决定和时间；
- 每一步错误和重试次数。

同一时间只允许一个运行中的测试。重置会创建新的运行 ID，不修改已经完成的历史报告。

## 内部页面

测试入口位于 Options 的内部工具区，不出现在普通 Studio 页面，不增加日常使用门槛。

页面包含三个阶段：

1. 准备：显示 24 条范围、当前模型、最低 240 次调用；若每个 Studio 组都需要修复，最多 288 次，不含 Provider 重试。
2. 运行：显示完成数量、当前阶段、失败数、重试状态和暂停/继续。
3. 审核与报告：一次性展示 10 条内容，接收总分和可选批注，随后展示汇总结果。

普通用户不会看到这些测试控件。

## 错误处理

- 缺少 API Key 或使用 mock key 时，在第一次调用前阻止测试。
- 开始时固定 Provider、模型和生成配置；中途改变配置会暂停并要求重新开始，避免混合模型污染数据。
- 限流和临时网络错误使用有上限的退避重试，并保留进度。
- JSON 错误、候选缺失和质量门控失败按素材与组记录，可单独重试。
- 页面关闭和浏览器重启不会丢失进度。
- 缺少任意必要输出、盲评或用户总分时，报告不能变成通过。
- 重置测试不能删除正常 Studio、Posts、Loop 或账号配置。

## 开关策略

将旧的共享开关拆分为：

`{ zhPostStudio: boolean, zhPostAuto: boolean }`

旧的 `zhPost` 不能证明授权的是哪个场景，因此升级时两个新开关都初始化为 `false`。

只有全部测试门槛通过后，当前安装才设置 `zhPostStudio: true`。本阶段永远不会修改 `zhPostAuto`，也不会影响 Reply。

失败或未完成时，两个模式保持关闭。报告列出最弱的内容类型和代表性问题，供下一版 Skill 迭代。

## 自动化测试

必须覆盖：

- 24 条素材按 6 类均衡选择；
- A/B 两组使用相同模型和上下文；
- 两组都运行真实 Studio 生产链路；
- 匿名排序与顺序互换评审；
- 平局和至少 18 条非平局门槛；
- 10 条审核集的类型覆盖和风险优先级；
- 总分与可选批注解析；
- 85 分审核通过线和 65% 模型胜率；
- 事实、安全和模板硬门槛；
- 检查点续跑不重复调用；
- API 缺失、限流、无效输出和配置变化；
- 审核集提升但隐藏集下降时阻止上线；
- benchmark 存储不污染 Loop 与发布数据；
- 通过后只开启 Studio，绝不开启 Auto 或 Reply；
- 现有多语言、Studio、Auto、Reply、隐私与存储回归。

## 当前发布决定

`zh-x-post@1.0.0` 继续保持关闭，直到真实批量盲测完成并通过。是否完成代码不是上线依据；真实审核数据和安全指标才是依据。
